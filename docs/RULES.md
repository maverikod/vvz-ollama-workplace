# Project rules (OLLAMA workstation)

**Author:** Vasiliy Zdanovskiy  
**Email:** vasilyvz@gmail.com  

Rules derived from [standards.md](standards.md) and [project_structure.md](project_structure.md). All contributors and AI assistants must follow these rules.

---

## 1. Structure and layout

- **Root layout:** Use the structure defined in `docs/project_structure.md`: `docs/`, `src/`, `tests/`, `config/`, `examples/`, `code_analysis/`, `.venv/`, `projectid`, `pyproject.toml` (or equivalent), `.gitignore`, `README.md`.
- **docs/** — Documentation and guides (документация и руководства). Hand-written specs, standards, how-tos, and reference docs. Language: English unless the product owner requests otherwise.
- **docs/reports/** — AI reports only (отчёты ИИ). Put all AI-generated reports, analyses, and run summaries here. Do not use this folder for hand-written documentation.
- **docs/plans/** — Plans (планы). Implementation plans, cleanup plans, roadmaps, and other planning documents.
- **config/** — Example and schema config files only. No hardcoded URLs or credentials in code; use config or environment.
- **code_analysis/** — Generated by code_mapper only; do not edit by hand. Run code_mapper after each batch of file changes.

---

## 2. Code and files

- **One class per file** (except small enums/exceptions). Split into facade + smaller modules when a class or file grows.
- **File size:** No file over **350–400 lines**. Split as soon as the limit is reached.
- **File header:** Every code and documentation file MUST have a header docstring with: **Author: Vasiliy Zdanovskiy**, **email: vasilyvz@gmail.com**.
- **Imports:** Only at the top of the file, except when implementing lazy loading.
- **Production code:** No `pass` instead of implementation; use `NotImplemented` only in abstract methods.

---

## 3. Quality checks

- After each change to a code file: run **black**, **flake8**, **mypy** and fix all reported issues.
- After each batch of file changes: run **code_mapper** so that `code_analysis/` indices stay up to date.

---

## 4. Environment and dependencies

- Use the project **.venv** in the project root. Do **not** use `--break-system-packages` for installing packages.
- Prefer **mcp-proxy-adapter** (client, Command, registry) for MCP Proxy calls; do not duplicate proxy protocol logic.

---

## 5. OLLAMA tools (workstation)

- Tools exposed to OLLAMA MUST use the format: `{ "type": "function", "function": { "name", "description", "parameters" } }` with JSON Schema for `parameters`.
- **Mandatory tools:** `list_servers`, `call_server`, `help`. Use parameter names aligned with the MCP Proxy API: `server_id`, `copy_number`, `command`, `params`; `page`, `page_size`, `filter_enabled` where applicable.

---

## 6. Configuration

- **Required:** `mcp_proxy_url`, `ollama_base_url`, `ollama_model`.
- **Optional:** `ollama_timeout`, `max_tool_rounds`, TLS/API keys as needed.
- Single source: config file (YAML/JSON) or environment variables.

---

## 7. Errors and logging

- On proxy call failure: put the **error message in the tool result content** so the model can see it; do not drop the conversation.
- Log tool invocations (tool name, arguments without secrets, proxy response status). Do not log sensitive data (tokens, keys, full message bodies).

---

## 8. Language

- **Documentation:** English (unless the product owner explicitly requests otherwise).
- **Code, comments, docstrings, tests:** English only.

---

## 9. Commits and indices

- After completing a logical change (single file or planned batch): make a **commit**. Push only when the user asks.
- Keep **code_analysis** indices up to date by running code_mapper after file changes.

---

## 10. Out of scope

- Do not modify files outside the project without the user’s permission.
- Project identifier for tools is in the **projectid** file in the project root.
