# ТЗ: подпроект model_workspace

**Author:** Vasiliy Zdanovskiy  
**email:** vasilyvz@gmail.com

Техническое задание на подпроект «рабочее место модели»: приложение, использующее клиенты redis-adapter и ollama-adapter. Общий контекст — в корне репозитория: `docs/plans/refactoring_adapter_structure/SPEC.md`.

---

## 1. Назначение

- **model-workspace-server** — приложение «рабочее место модели»: оркестрация чата с моделью, сессии, контекст, вызов инструментов через MCP Proxy.
- **Внутри контейнера/процесса нет Redis и Ollama.** Все обращения к хранилищу и к модели — через **клиента redis-adapter** и **клиента ollama-adapter** (по WebSocket через прокси).
- Тоннельный режим: вызывающий код использует API Redis и Ollama через эти клиенты; трафик идёт через адаптер и прокси.

---

## 2. Зависимости от подпроектов

- **redis-adapter** (пакет `redis_adapter` или `redis-adapter`) — клиент Redis на базе адаптера: `execute(command, *args)` и обёртки (get, set, hgetall, …).
- **ollama-adapter** (пакет `ollama_adapter` или `ollama-adapter`) — клиент Ollama на базе адаптера: `execute(method, **params)` и обёртки (tags, chat, generate, …).
- Подключение к серверам redis-adapter и ollama-adapter — по **WebSocket** (через MCP Proxy); конфигурация (URL прокси, идентификаторы серверов) — в конфиге подпроекта.

---

## 3. Конфигурация

- Собственные конфиги в каталоге подпроекта (`config/`): URL прокси, параметры подключения к redis-adapter и ollama-adapter, модель по умолчанию, лимиты, политики команд и т.д.
- Без хардкода адресов и учётных данных; при необходимости — ссылка на общие сертификаты (mtls_certificates в корне репозитория).

---

## 4. Контейнер

- Контейнер **model-workspace-server**: только приложение данного подпроекта + зависимости (в т.ч. клиенты redis-adapter и ollama-adapter). Ни Redis, ни Ollama внутри образа не запускаются.

---

## 5. Функциональность (обзор)

- Инициализация сессии, чат с моделью (через клиент ollama-adapter), сохранение/загрузка сообщений (через клиент redis-adapter при необходимости).
- Вызов инструментов через MCP Proxy (list_servers, call_server, help и т.д.), встраивание ответов в контекст модели.
- Регистрация/интеграция с прокси по требованиям общего ТЗ (если приложение само регистрируется как сервис).

Детализация сценариев и API — в отдельной спецификации или в общем SPEC в корне.
